# 系统管理模块

## 1. 模块定位

系统管理负责平台登录、用户身份、角色权限、系统配置和基础审计。它不负责企业组织同步、复杂审批流和多租户隔离，这些能力作为长期平台治理能力独立演进。

## 2. 职责边界

| 负责 | 不负责 |
|------|--------|
| 用户登录、退出、当前用户信息 | 企业统一身份同步 |
| 平台角色：管理员、项目负责人、项目成员 | 复杂组织树和岗位体系 |
| 基础认证拦截和接口权限判断 | 数据脱敏审批流 |
| 创建人、更新时间等审计字段 | 完整审计报表和风控模型 |
| 系统参数读取和维护 | 分布式配置中心 |

## 3. 需求规格

### REQ-SYS-001 用户登录和角色

**用户故事**：作为平台管理员，我希望平台具备基础登录和角色控制，避免匿名访问测试资产。

**功能说明**：

1. 支持用户名密码登录。
2. MVP 内置管理员、项目负责人、项目成员三类角色。
3. 登录后可查看当前用户信息和权限范围。
4. 业务接口必须能识别当前用户，写操作必须落审计字段。

**输入/输出**：

| 类型 | 内容 |
|------|------|
| 输入 | 用户名、密码 |
| 输出 | 登录状态、用户信息、角色信息 |

**异常场景**：

1. 用户名或密码错误时拒绝登录。
2. 禁用用户无法登录。
3. 未登录访问业务接口时返回认证失败。

**验收标准**：

1. 合法用户可以登录平台。
2. 错误密码无法登录。
3. 未登录用户无法访问项目、脚本、任务和报告页面。

### REQ-SYS-002 基础审计字段

**用户故事**：作为平台管理员，我希望关键数据记录创建人和更新时间，便于后续排查问题。

**功能说明**：

1. MVP 核心表统一记录创建人、创建时间、更新人、更新时间。
2. 关键操作记录操作人，包括项目归档、脚本上传、任务提交、任务取消、报告生成。
3. 审计字段不允许普通用户手工修改。

**输入/输出**：

| 类型 | 内容 |
|------|------|
| 输入 | 当前登录用户、业务操作 |
| 输出 | 审计字段、操作记录 |

**异常场景**：

1. 无登录用户上下文时拒绝执行业务写操作。
2. 系统自动任务使用系统用户标识记录。

**验收标准**：

1. 新建项目、上传脚本、提交任务和生成报告后能看到创建人和创建时间。
2. 取消任务后能看到取消操作人和取消时间。
3. 用户无法通过普通编辑接口篡改审计字段。

## 4. 关键实体

| 实体 | 关键字段 | 说明 |
|------|----------|------|
| `user` | `id`, `username`, `password_hash`, `display_name`, `status`, `created_at` | 平台用户 |
| `role` | `id`, `code`, `name` | 系统角色 |
| `user_role` | `user_id`, `role_id` | 用户角色关系 |
| `system_config` | `config_key`, `config_value`, `description`, `updated_at` | 系统配置 |
| `operation_log` | `id`, `operator_id`, `action`, `target_type`, `target_id`, `created_at` | 关键操作日志 |

## 5. 接口草案

| 方法 | 路径 | 说明 |
|------|------|------|
| `POST` | `/api/auth/login` | 用户登录 |
| `POST` | `/api/auth/logout` | 用户退出 |
| `GET` | `/api/auth/me` | 当前用户信息 |
| `GET` | `/api/system/users` | 用户列表 |
| `POST` | `/api/system/users` | 创建用户 |
| `PATCH` | `/api/system/users/{id}/status` | 启用或禁用用户 |
| `GET` | `/api/system/configs` | 系统配置列表 |
| `PATCH` | `/api/system/configs/{key}` | 更新系统配置 |

## 6. 详细设计调整点

1. 当前阶段可以使用轻量登录和服务端会话模型，但接口层必须统一从认证上下文获取用户，不允许业务方法传入伪造用户名。
2. 权限判断先落到 `PermissionService`，由它组合系统角色和项目成员关系，避免权限逻辑散落在 Controller。
3. 审计字段作为领域服务写入职责，不交给前端传递。
4. `operation_log` 首版只记录关键动作，不做全量字段 diff。
5. 后续接入 Spring Security 时保持登录响应和权限错误结构不变，降低前端改造成本。
