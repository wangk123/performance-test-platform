# 测试执行模块

## 1. 模块定位

测试执行负责把脚本版本和执行配置转化为一次可追踪的压测运行过程。MVP 使用单机队列，长期通过 `TaskScheduler` 和 `TestExecutor` 抽象演进到分布式执行。

## 2. 职责边界

| 负责 | 不负责 |
|------|--------|
| 测试任务创建和执行配置快照 | 脚本文件上传 |
| 单机队列和状态流转 | 报告页面展示 |
| JMeter 启动、停止、日志归档 | 指标图表渲染 |
| 执行结果文件登记 | 监控指标采集 |

## 3. 需求规格

### REQ-EXEC-001 提交单机执行任务

**用户故事**：作为性能测试工程师，我希望选择脚本版本并提交测试任务，让平台在服务器上执行 JMeter。

**功能说明**：

1. 支持选择项目、脚本版本和执行配置后提交任务。
2. MVP 使用单机队列，同一时刻默认只运行有限数量任务，具体上限由系统配置控制。
3. 任务状态包括待执行、运行中、成功、失败、已取消、异常中断。
4. 每次提交生成测试任务和执行记录，执行配置固化保存。

**输入/输出**：

| 类型 | 内容 |
|------|------|
| 输入 | 项目 ID、脚本版本 ID、执行配置、任务名称 |
| 输出 | 测试任务 ID、执行记录 ID、任务状态 |

**异常场景**：

1. 脚本版本不可用时拒绝提交。
2. 队列达到上限时任务保持待执行状态。
3. JMeter 启动失败时任务进入失败状态，并记录错误日志。

**验收标准**：

1. 用户可以提交测试任务，并在任务列表看到待执行或运行中状态。
2. 任务执行配置在提交后被固化，后续脚本参数变化不影响该任务。
3. JMeter 启动失败时，任务详情显示失败原因。

### REQ-EXEC-002 查看执行状态和日志

**用户故事**：作为性能测试工程师，我希望实时查看任务状态和日志，判断压测是否正常。

**功能说明**：

1. 支持任务列表按项目、状态、创建人、时间范围筛选。
2. 支持任务详情展示状态、开始时间、结束时间、运行时长、执行参数、错误摘要。
3. 支持查看 JMeter 执行日志和平台调度日志。
4. 页面刷新后仍可恢复查询最新状态。

**输入/输出**：

| 类型 | 内容 |
|------|------|
| 输入 | 任务 ID、筛选条件 |
| 输出 | 任务列表、任务详情、执行日志 |

**异常场景**：

1. 任务不存在时返回明确错误。
2. 日志文件丢失时显示日志不可用，不影响任务状态查看。
3. 平台重启后运行中任务标记为异常中断。

**验收标准**：

1. 用户可以按状态筛选任务列表。
2. 运行中任务详情能看到开始时间、运行时长和执行参数。
3. 平台重启后，原运行中任务不会长期停留在运行中状态。

### REQ-EXEC-003 取消任务

**用户故事**：作为性能测试工程师，我希望停止误提交或已经不需要的压测任务，避免持续占用资源。

**功能说明**：

1. 支持取消待执行和运行中的任务。
2. 待执行任务取消后不再进入执行队列。
3. 运行中任务取消时，平台尝试停止对应 JMeter 进程。
4. 取消操作记录操作人和操作时间。

**输入/输出**：

| 类型 | 内容 |
|------|------|
| 输入 | 任务 ID、取消原因 |
| 输出 | 取消结果、任务状态 |

**异常场景**：

1. 已结束任务不允许取消。
2. JMeter 进程停止失败时，任务进入失败或异常中断状态，并记录原因。

**验收标准**：

1. 待执行任务取消后状态变为已取消。
2. 运行中任务取消后不再继续生成新的采样结果。
3. 取消操作在任务详情中可追溯。

## 4. 关键实体

| 实体 | 关键字段 | 说明 |
|------|----------|------|
| `test_task` | `id`, `project_id`, `script_version_id`, `name`, `status`, `created_by`, `created_at` | 测试任务 |
| `task_execution` | `id`, `task_id`, `config_json`, `status`, `start_time`, `end_time`, `duration_ms`, `cancel_by`, `cancel_reason` | 执行记录 |
| `execution_result` | `id`, `execution_id`, `result_file_path`, `log_file_path`, `sample_count`, `error_count`, `summary_json` | 执行结果 |

状态流转：

```text
QUEUED -> RUNNING -> SUCCESS
QUEUED -> RUNNING -> FAILED
QUEUED -> CANCELLED
RUNNING -> CANCELLED
RUNNING -> INTERRUPTED
```

## 5. 接口草案

| 方法 | 路径 | 说明 |
|------|------|------|
| `POST` | `/api/projects/{projectId}/tasks` | 提交测试任务 |
| `GET` | `/api/projects/{projectId}/tasks` | 任务列表 |
| `GET` | `/api/tasks/{taskId}` | 任务详情 |
| `GET` | `/api/tasks/{taskId}/logs` | 执行日志 |
| `POST` | `/api/tasks/{taskId}/cancel` | 取消任务 |
| `GET` | `/api/executions/{executionId}/result` | 执行结果摘要 |

## 6. 详细设计调整点

1. `test_task` 代表用户提交意图，`task_execution` 代表实际执行实例；首版一任务一执行，后续可支持重跑。
2. 执行配置必须保存为 JSON 快照，字段至少包括线程数、循环次数、持续时间、Ramp-Up、目标环境、JMeter 属性。
3. 执行目录按 `storage/executions/{projectId}/{taskId}/{executionId}` 隔离，目录内放置脚本副本、日志、结果文件。
4. 调度器启动时扫描 `RUNNING` 状态记录，无法确认存活的任务标记为 `INTERRUPTED`。
5. JMeter API 和 CLI 先通过 `TestExecutor` 隔离，避免执行方式变化影响任务接口。
