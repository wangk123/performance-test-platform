# 性能测试平台 - 需求规格与架构设计方案

## 一、产品定位

性能测试平台面向内部测试团队和项目研发团队，提供从性能脚本管理、测试执行、过程状态查看到测试报告生成的统一工作台。第一版不追求一次性覆盖完整性能工程体系，而是优先跑通一次可复用、可追踪、可交付的压测闭环。

### 1.1 建设目标

1. 降低 JMeter 压测脚本执行和结果整理的门槛。
2. 将测试项目、脚本版本、执行配置、执行记录和报告沉淀为可追溯资产。
3. 为后续监控采集、造数工厂、函数库、Git 关联和 AI 分析预留清晰扩展点。

### 1.2 目标用户

| 角色 | 主要诉求 | 典型操作 |
|------|----------|----------|
| 测试负责人 | 统一管理项目和测试资产，查看报告结论 | 创建项目、分配成员、查看报告 |
| 性能测试工程师 | 上传脚本、配置参数、发起执行、分析结果 | 管理 JMX、提交任务、下载报告 |
| 研发工程师 | 查看压测结果和错误信息，辅助定位问题 | 查看执行结果、下载报告 |
| 平台管理员 | 管理用户、角色、系统配置和运行风险 | 用户管理、权限配置、系统配置 |

### 1.3 MVP 目标

MVP 只覆盖核心闭环：

```text
创建项目 -> 上传 JMX 脚本 -> 配置执行参数 -> 提交单机执行
-> 查看任务状态 -> 生成基础报告 -> 下载或分享报告
```

MVP 成功标准：

1. 用户可以在平台内完成一次 JMeter 压测任务。
2. 每次执行都能关联项目、脚本版本、执行配置、执行日志和测试报告。
3. 报告至少包含 TPS、响应时间、错误率、采样统计和失败摘要。
4. 长任务状态在页面刷新后仍可恢复查询。

### 1.4 非目标范围

以下能力不进入 MVP 必做范围：

1. 分布式压测执行。
2. RabbitMQ、Kafka、Redis 等外部中间件依赖。
3. 服务器监控 Agent、Prometheus、Grafana 集成。
4. AI 自动分析、Git 代码关联、日志智能分析。
5. 造数工厂、自定义函数市场、辅助脚本编排。
6. 多租户、复杂组织架构、精细化数据权限。

## 二、核心用户旅程

### 2.1 一次压测任务的主流程

1. 测试负责人创建性能测试项目，并添加项目成员。
2. 性能测试工程师上传 JMeter JMX 文件，平台生成脚本版本。
3. 工程师填写执行配置，包括线程数、循环次数、持续时间、Ramp-Up、目标环境和备注。
4. 工程师提交测试任务，平台进入单机执行队列。
5. 平台调用 JMeter API 启动任务，并持续记录任务状态、执行日志和结果文件。
6. 用户在任务详情页查看状态、开始时间、运行时长、错误数和实时摘要。
7. 任务结束后，平台解析结果文件并生成基础测试报告。
8. 用户查看报告，下载 Markdown 或 HTML 报告，并将报告链接分享给项目成员。

### 2.2 异常流程

| 场景 | 平台行为 | 用户可见结果 |
|------|----------|--------------|
| JMX 文件格式错误 | 拒绝上传或标记解析失败 | 显示失败原因，不生成脚本版本 |
| 执行参数不合法 | 阻止提交任务 | 标出错误字段和原因 |
| JMeter 启动失败 | 任务进入失败状态 | 记录错误日志和失败原因 |
| 用户取消任务 | 尝试停止 JMeter 进程 | 任务状态变为已取消，保留已产生数据 |
| 平台重启 | 从数据库恢复任务状态 | 已结束任务可查看，运行中任务标记为异常中断 |
| 报告生成失败 | 执行记录保留，报告状态失败 | 用户可重新生成报告 |

## 三、功能范围

### 3.1 MVP 必做模块

| 模块 | 范围 | 说明 |
|------|------|------|
| 项目管理 | 项目 CRUD、归档、成员管理、基础权限 | 平台资产的组织入口 |
| 脚本管理 | JMX 上传、脚本版本、参数配置、导入导出 | 管理可执行测试脚本 |
| 测试执行 | 单机队列、执行配置、状态查询、取消、失败记录 | 跑通压测任务 |
| 报告管理 | 基础指标统计、Markdown/HTML 报告、下载和分享 | 形成交付物 |
| 系统管理 | 登录、用户、角色、基础审计字段 | 支撑内部使用 |

### 3.2 Phase 2 增强模块

| 模块 | 范围 | 进入条件 |
|------|------|----------|
| 监控采集 | 服务器资源采集、趋势图、报告整合 | MVP 执行闭环稳定后 |
| 报告增强 | Word/PDF 导出、多次报告对比 | 基础报告被实际使用后 |
| 日志检索 | 日志上传、关键字检索、失败摘要 | 用户需要定位执行失败时 |
| 辅助脚本 | 前置/后置 Shell 或 Python 脚本 | 测试准备流程重复度较高时 |

### 3.3 远期扩展模块

| 模块 | 范围 | 说明 |
|------|------|------|
| 造数工厂 | 数据模板、生成规则、CSV/JSON/SQL 导出 | 提升数据准备效率 |
| 函数库 | Groovy 函数管理、调试、分类 | 服务复杂 JMeter 脚本 |
| Git 管理 | 仓库配置、代码拉取、提交记录查看 | 为代码关联分析做准备 |
| AI 分析 | 多模型适配、Prompt 模板、优化建议 | 建立在稳定数据和报告之上 |
| 分布式执行 | 多节点调度、执行协调、结果聚合 | 执行规模超过单机能力后 |

## 四、MVP 需求规格

需求条目统一格式：

- 编号：`REQ-模块-序号`
- 用户故事：谁在什么场景下要完成什么目标。
- 功能说明：平台需要提供的能力。
- 输入/输出：关键输入和系统输出。
- 异常场景：必须处理的失败或边界情况。
- 验收标准：可验证的完成条件。

### 4.1 项目管理

#### REQ-PROJ-001 创建和维护项目

**用户故事**：作为测试负责人，我希望为每个被测系统创建独立项目，以便集中管理脚本、任务和报告。

**功能说明**：

1. 支持创建、编辑、查询、归档和恢复项目。
2. 项目字段包括项目名称、项目编码、描述、负责人、状态、创建时间、更新时间。
3. 项目编码在系统内唯一，归档项目默认不出现在新建任务选择列表中。

**输入/输出**：

| 类型 | 内容 |
|------|------|
| 输入 | 项目名称、项目编码、描述、负责人 |
| 输出 | 项目详情、项目列表、操作结果 |

**异常场景**：

1. 项目编码重复时拒绝保存。
2. 项目名称为空时拒绝保存。
3. 已有关联任务的项目只允许归档，不允许物理删除。

**验收标准**：

1. 用户可以创建一个项目，并在项目列表中看到该项目。
2. 重复项目编码无法保存，并显示明确错误信息。
3. 归档项目后，该项目不再出现在新建测试任务的项目选择项中。

#### REQ-PROJ-002 成员与基础权限

**用户故事**：作为测试负责人，我希望控制项目成员范围，避免无关用户修改测试资产。

**功能说明**：

1. 支持为项目添加和移除成员。
2. MVP 提供三类角色：管理员、项目负责人、项目成员。
3. 管理员可管理所有项目；项目负责人可管理自己负责的项目；项目成员可查看项目和提交测试任务。

**输入/输出**：

| 类型 | 内容 |
|------|------|
| 输入 | 项目 ID、用户 ID、项目角色 |
| 输出 | 成员列表、权限校验结果 |

**异常场景**：

1. 移除项目负责人前必须先指定新的负责人。
2. 非项目成员访问项目详情时返回无权限。

**验收标准**：

1. 项目负责人可以添加项目成员。
2. 非项目成员无法查看该项目下的脚本、任务和报告。
3. 管理员可以查看和维护所有项目。

### 4.2 脚本管理

#### REQ-SCRIPT-001 上传 JMX 脚本

**用户故事**：作为性能测试工程师，我希望上传 JMeter JMX 文件，作为后续执行任务的脚本来源。

**功能说明**：

1. 支持在项目下上传 `.jmx` 文件。
2. 上传后生成脚本记录和脚本版本。
3. 记录文件名、文件大小、文件哈希、上传人、上传时间、版本号和存储路径。
4. 同一脚本再次上传时生成新版本，不覆盖历史版本。

**输入/输出**：

| 类型 | 内容 |
|------|------|
| 输入 | 项目 ID、JMX 文件、脚本名称、备注 |
| 输出 | 脚本详情、版本号、上传结果 |

**异常场景**：

1. 文件扩展名不是 `.jmx` 时拒绝上传。
2. 文件超过系统限制时拒绝上传。
3. JMX 解析失败时保存失败原因，脚本版本状态为解析失败。

**验收标准**：

1. 用户可以上传合法 JMX 文件，并在脚本列表看到脚本和版本号。
2. 同一脚本上传第二次后，版本号递增，旧版本仍可查看。
3. 非 JMX 文件无法上传，并提示文件类型不支持。

#### REQ-SCRIPT-002 配置脚本参数

**用户故事**：作为性能测试工程师，我希望在执行前配置线程数、循环次数等参数，减少直接修改 JMX 文件的频率。

**功能说明**：

1. 支持为脚本版本维护默认执行参数。
2. MVP 参数包括线程数、循环次数、持续时间、Ramp-Up、目标环境、JMeter 属性扩展项。
3. 执行任务可复制脚本默认参数并允许本次执行覆盖。

**输入/输出**：

| 类型 | 内容 |
|------|------|
| 输入 | 脚本版本 ID、参数键值、参数说明 |
| 输出 | 参数列表、参数校验结果 |

**异常场景**：

1. 线程数、循环次数、持续时间不能为负数。
2. 参数键重复时拒绝保存。
3. 已执行任务的历史配置不随脚本默认参数变化而改变。

**验收标准**：

1. 用户可以为脚本版本保存默认执行参数。
2. 新建任务时自动带出默认参数。
3. 修改脚本默认参数不会改变已存在执行记录中的执行配置。

### 4.3 测试执行

#### REQ-EXEC-001 提交单机执行任务

**用户故事**：作为性能测试工程师，我希望选择脚本版本并提交测试任务，让平台在服务器上执行 JMeter。

**功能说明**：

1. 支持选择项目、脚本版本和执行配置后提交任务。
2. MVP 使用单机队列，同一时刻默认只运行有限数量任务，具体上限由系统配置控制。
3. 任务状态包括待执行、运行中、成功、失败、已取消、异常中断。
4. 每次提交生成测试任务和执行记录，执行配置固化保存。

**输入/输出**：

| 类型 | 内容 |
|------|------|
| 输入 | 项目 ID、脚本版本 ID、执行配置、任务名称 |
| 输出 | 测试任务 ID、执行记录 ID、任务状态 |

**异常场景**：

1. 脚本版本不可用时拒绝提交。
2. 队列达到上限时任务保持待执行状态。
3. JMeter 启动失败时任务进入失败状态，并记录错误日志。

**验收标准**：

1. 用户可以提交测试任务，并在任务列表看到待执行或运行中状态。
2. 任务执行配置在提交后被固化，后续脚本参数变化不影响该任务。
3. JMeter 启动失败时，任务详情显示失败原因。

#### REQ-EXEC-002 查看执行状态和日志

**用户故事**：作为性能测试工程师，我希望实时查看任务状态和日志，判断压测是否正常。

**功能说明**：

1. 支持任务列表按项目、状态、创建人、时间范围筛选。
2. 支持任务详情展示状态、开始时间、结束时间、运行时长、执行参数、错误摘要。
3. 支持查看 JMeter 执行日志和平台调度日志。
4. 页面刷新后仍可恢复查询最新状态。

**输入/输出**：

| 类型 | 内容 |
|------|------|
| 输入 | 任务 ID、筛选条件 |
| 输出 | 任务列表、任务详情、执行日志 |

**异常场景**：

1. 任务不存在时返回明确错误。
2. 日志文件丢失时显示日志不可用，不影响任务状态查看。
3. 平台重启后运行中任务标记为异常中断。

**验收标准**：

1. 用户可以按状态筛选任务列表。
2. 运行中任务详情能看到开始时间、运行时长和执行参数。
3. 平台重启后，原运行中任务不会长期停留在运行中状态。

#### REQ-EXEC-003 取消任务

**用户故事**：作为性能测试工程师，我希望停止误提交或已经不需要的压测任务，避免持续占用资源。

**功能说明**：

1. 支持取消待执行和运行中的任务。
2. 待执行任务取消后不再进入执行队列。
3. 运行中任务取消时，平台尝试停止对应 JMeter 进程。
4. 取消操作记录操作人和操作时间。

**输入/输出**：

| 类型 | 内容 |
|------|------|
| 输入 | 任务 ID、取消原因 |
| 输出 | 取消结果、任务状态 |

**异常场景**：

1. 已结束任务不允许取消。
2. JMeter 进程停止失败时，任务进入失败或异常中断状态，并记录原因。

**验收标准**：

1. 待执行任务取消后状态变为已取消。
2. 运行中任务取消后不再继续生成新的采样结果。
3. 取消操作在任务详情中可追溯。

### 4.4 报告管理

#### REQ-REPORT-001 生成基础测试报告

**用户故事**：作为性能测试工程师，我希望任务结束后自动生成报告，用于交付测试结果。

**功能说明**：

1. 任务成功、失败或已取消后，平台可基于结果文件生成报告。
2. 报告包含任务信息、执行配置、聚合指标、接口明细、错误摘要和结论备注。
3. 聚合指标至少包括 TPS、平均响应时间、P90、P95、P99、最大响应时间、最小响应时间、错误率、采样总数。
4. MVP 支持 Markdown 和 HTML 两种报告查看或下载方式。

**输入/输出**：

| 类型 | 内容 |
|------|------|
| 输入 | 执行记录 ID、报告标题、结论备注 |
| 输出 | 测试报告、报告文件、报告链接 |

**异常场景**：

1. 结果文件不存在时报告生成失败并提示原因。
2. 结果文件过大时异步生成报告，并展示生成状态。
3. 报告生成失败时允许重新生成。

**验收标准**：

1. 已结束任务可以生成一份基础测试报告。
2. 报告中能看到 TPS、响应时间分位值、错误率和采样总数。
3. 用户可以下载 Markdown 或 HTML 报告文件。

#### REQ-REPORT-002 报告分享和追溯

**用户故事**：作为测试负责人，我希望报告可以被项目成员访问，并能追溯到原始任务和脚本版本。

**功能说明**：

1. 报告与项目、任务、执行记录、脚本版本关联。
2. 项目成员可查看项目内报告。
3. 报告页面提供原始任务入口和脚本版本信息。

**输入/输出**：

| 类型 | 内容 |
|------|------|
| 输入 | 报告 ID、访问用户 |
| 输出 | 报告详情、权限校验结果 |

**异常场景**：

1. 非项目成员访问报告时返回无权限。
2. 原始任务被归档时报告仍可查看。

**验收标准**：

1. 项目成员可以通过报告链接查看报告。
2. 报告详情能跳转回对应任务详情。
3. 非项目成员无法查看报告内容。

### 4.5 系统管理

#### REQ-SYS-001 用户登录和角色

**用户故事**：作为平台管理员，我希望平台具备基础登录和角色控制，避免匿名访问测试资产。

**功能说明**：

1. 支持用户名密码登录。
2. MVP 内置管理员、项目负责人、项目成员三类角色。
3. 登录后可查看当前用户信息和权限范围。

**输入/输出**：

| 类型 | 内容 |
|------|------|
| 输入 | 用户名、密码 |
| 输出 | 登录状态、用户信息、角色信息 |

**异常场景**：

1. 用户名或密码错误时拒绝登录。
2. 禁用用户无法登录。
3. 未登录访问业务接口时返回认证失败。

**验收标准**：

1. 合法用户可以登录平台。
2. 错误密码无法登录。
3. 未登录用户无法访问项目、脚本、任务和报告页面。

#### REQ-SYS-002 基础审计字段

**用户故事**：作为平台管理员，我希望关键数据记录创建人和更新时间，便于后续排查问题。

**功能说明**：

1. MVP 核心表统一记录创建人、创建时间、更新人、更新时间。
2. 关键操作记录操作人，包括项目归档、任务提交、任务取消、报告生成。
3. 审计字段不允许普通用户手工修改。

**输入/输出**：

| 类型 | 内容 |
|------|------|
| 输入 | 当前登录用户、业务操作 |
| 输出 | 审计字段、操作记录 |

**异常场景**：

1. 无登录用户上下文时拒绝执行业务写操作。
2. 系统自动任务使用系统用户标识记录。

**验收标准**：

1. 新建项目、上传脚本、提交任务和生成报告后能看到创建人和创建时间。
2. 取消任务后能看到取消操作人和取消时间。
3. 用户无法通过普通编辑接口篡改审计字段。

## 五、需求驱动架构

### 5.1 整体架构风格

MVP 采用单体分层架构和模块化设计。这样可以降低部署复杂度，优先验证业务闭环，同时通过接口抽象保留后续替换空间。

```text
┌─────────────────────────────────────────┐
│ 前端层 Vue 3 + TypeScript + Element Plus │
│ 项目、脚本、任务、报告、系统管理页面       │
└─────────────────────────────────────────┘
                  ↓ HTTP / WebSocket
┌─────────────────────────────────────────┐
│ 应用层 Spring Boot 3                    │
│ 认证鉴权、业务编排、事务控制、状态推送     │
└─────────────────────────────────────────┘
                  ↓
┌─────────────────────────────────────────┐
│ 领域层 Domain                           │
│ 项目、脚本、执行、报告、用户权限           │
└─────────────────────────────────────────┘
                  ↓
┌─────────────────────────────────────────┐
│ 基础设施层 Infrastructure               │
│ 数据库、文件存储、JMeter、缓存、报告渲染   │
└─────────────────────────────────────────┘
                  ↓
┌─────────────────────────────────────────┐
│ 外部依赖                                │
│ MySQL/H2、文件系统、JMeter Engine        │
└─────────────────────────────────────────┘
```

### 5.2 技术选型

| 分类 | MVP 选型 | 需求依据 | 后续扩展 |
|------|----------|----------|----------|
| 后端框架 | Spring Boot 3.x | 单体分层、事务控制、JMeter Java 生态 | Spring Cloud 微服务 |
| 前端框架 | Vue 3 + TypeScript | 复杂表单、状态页面、类型约束 | 微前端 |
| UI 组件库 | Element Plus | 企业后台组件完整 | Ant Design Vue |
| 数据可视化 | ECharts | 报告和结果图表 | D3.js |
| 数据库 | H2 开发 / MySQL 生产 | MVP 数据关系明确，部署简单 | 分库分表 |
| 文件存储 | 本地文件系统 | 存储 JMX、日志、结果、报告 | MinIO、OSS、COS |
| 缓存 | Caffeine | 轻量缓存用户权限和配置 | Redis |
| 任务调度 | Spring Async + 数据库任务表 | 单机执行队列足够支撑 MVP | RabbitMQ、Kafka |
| 测试引擎 | JMeter API | 直接执行 JMX，符合现有能力 | Gatling、K6 |
| 报告生成 | Markdown + HTML | 易读、易下载、易生成 | 模板引擎、Word、PDF |

### 5.3 MVP 接口抽象

| 能力 | 接口 | MVP 实现 | 扩展方向 |
|------|------|----------|----------|
| 文件存储 | `FileStorage` | 本地文件系统 | MinIO、OSS、COS |
| 任务调度 | `TaskScheduler` | 数据库状态 + Spring Async | MQ、分布式调度 |
| 测试执行 | `TestExecutor` | JMeter API | Gatling、K6 |
| 报告生成 | `ReportGenerator` | Markdown/HTML 生成器 | Freemarker、Thymeleaf |
| 权限校验 | `PermissionService` | 基于角色和项目成员 | 组织架构、细粒度权限 |

暂不在 MVP 中实现 Redis、MQ、分布式协调、AI 模型适配、Git API 适配等接口。相关能力只保留需求和架构扩展说明，避免首版复杂度膨胀。

### 5.4 模块边界

| 模块 | 负责 | 不负责 |
|------|------|--------|
| Project | 项目、成员、项目状态 | 用户账号生命周期 |
| Script | JMX 文件、脚本版本、默认参数 | 实际任务执行 |
| Execution | 任务提交、队列、JMeter 调用、状态 | 报告展示和下载 |
| Report | 结果解析、指标聚合、报告文件 | 启停 JMeter |
| System | 用户、角色、基础权限、审计字段 | 企业组织同步 |

## 六、核心数据模型

### 6.1 实体关系

```text
User 1..n ProjectMember n..1 Project
Project 1..n Script 1..n ScriptVersion
ScriptVersion 1..n TestTask 1..n TaskExecution 1..1 ExecutionResult
```

| `user_role` | `user_id`, `role_id` | 用户角色关系 |
| `project` | `id`, `code`, `name`, `owner_id`, `status`, `description`, `created_by`, `created_at`, `updated_at` | 项目主表 |
| `project_member` | `project_id`, `user_id`, `project_role`, `created_at` | 项目成员 |

#### 脚本与执行

| 表 | 关键字段 | 说明 |
|----|----------|------|
| `script` | `id`, `project_id`, `name`, `status`, `created_by`, `created_at` | 脚本主表 |
| `script_version` | `id`, `script_id`, `version_no`, `file_name`, `file_hash`, `file_path`, `parse_status`, `remark`, `created_at` | 脚本版本 |

#### 报告与配置

| 表 | 关键字段 | 说明 |
|----|----------|------|
| `test_report` | `id`, `execution_id`, `title`, `format`, `file_path`, `status`, `summary`, `created_by`, `created_at` | 测试报告 |
| `system_config` | `config_key`, `config_value`, `description`, `updated_at` | 系统配置 |
| `operation_log` | `id`, `operator_id`, `action`, `target_type`, `target_id`, `created_at` | 关键操作日志 |

### 6.3 数据一致性要求

1. 测试任务提交后，执行配置以 JSON 快照固化在 `task_execution.config_json`。
2. 脚本版本文件不可被覆盖，只允许追加新版本。
3. 项目归档不删除脚本、任务、执行记录和报告。
4. 报告生成失败不影响执行记录查询。
5. 运行中任务在平台启动时需要校准状态，无法确认进程存活时标记为异常中断。

## 七、分阶段开发计划

长期平台采用 Spring Boot 精简单体起步，先把核心压测闭环做扎实，再按使用频率和工程风险逐步扩展。阶段推进遵循两个原则：

1. 每个阶段必须能独立交付可用能力，不能只交付技术底座。
2. 后一阶段只能依赖前一阶段稳定产出的数据资产和接口抽象，避免提前引入 Spring Cloud、MQ、Redis、分布式调度等重组件。

### 7.1 阶段总览

| 阶段 | 建议周期 | 核心目标 | 平台成熟度 |
|------|----------|----------|------------|
| Phase 0 | 1 周 | 原型验证和工程准备 | 明确边界，降低启动风险 |
| Phase 1 | 2-3 周 | 平台骨架、登录权限、项目管理 | 可登录、可管项目 |
| Phase 2 | 3-4 周 | 脚本管理和单机执行闭环 | 可完成一次 JMeter 压测 |
| Phase 3 | 2-3 周 | 报告和结果可视化 | 可交付基础报告 |
| Phase 4 | 3-4 周 | 监控采集和报告增强 | 可关联资源指标分析 |
| Phase 5 | 3-4 周 | 测试准备效率工具 | 可沉淀数据和函数资产 |
| Phase 6 | 4-6 周 | Git、日志和 AI 分析 | 可辅助定位瓶颈 |
| Phase 7 | 4-8 周 | 执行规模扩展 | 支持多节点和更高并发 |

周期为初始估算，实际排期以团队人数、JMeter 脚本复杂度、基础设施权限和使用反馈为准。

### Phase 0：原型验证和工程准备

**目标**：确认长期平台路线可行，锁定技术基线和开发规范，避免进入开发后频繁改底层方向。

**功能范围**：

1. 验证 Spring Boot 调用 JMeter API 或 JMeter CLI 的可行性。
2. 验证 JMX 上传、结果文件解析、报告指标计算的最小链路。
3. 明确单机执行的进程管理、目录结构、日志路径和结果文件格式。
4. 确定前后端仓库结构、模块命名、接口返回格式和错误码规范。

**技术重点**：

1. Spring Boot 3.x、Java 17 或以上版本。
2. Maven 多模块或单仓分层结构。
3. Vue 3、TypeScript、Vite、Element Plus。
4. JMeter API 优先；如果 API 集成复杂度过高，保留 JMeter CLI 作为执行适配器实现。

**交付物**：

1. 可运行的后端 Demo，能触发一次本地 JMeter 执行。
2. 可解析结果文件并输出 TPS、响应时间、错误率的验证代码。
3. 工程目录规范、配置规范、日志规范。
4. 第一版数据库命名和审计字段规范。

**验收重点**：

1. 本地环境可以通过一个命令启动后端验证程序。
2. 示例 JMX 能执行完成，并生成可解析的结果文件。
3. 团队确认继续采用 Spring Boot 精简单体路线。

**进入下一阶段条件**：

1. JMeter 执行方式已确定。
2. 文件目录、日志目录、结果目录已约定。
3. 前后端基础技术栈不再摇摆。

### Phase 1：平台骨架、登录权限与项目管理

**目标**：建立可运行的前后端基础框架和项目资产入口。

**功能范围**：

1. 用户登录、退出、当前用户信息查询。
2. 管理员、项目负责人、项目成员三类角色。
3. 项目创建、编辑、归档、恢复、成员维护。
4. 基础菜单、路由守卫、权限拦截。
5. 核心表的创建人、创建时间、更新人、更新时间。

**技术重点**：

1. Spring Boot 单体分层：controller、application、domain、infrastructure。
2. Spring Security 或轻量认证拦截器，优先满足内部平台登录和权限控制。
3. H2 开发库和 MySQL 生产库双配置。
4. Flyway 或 Liquibase 管理数据库变更。
5. 统一响应结构、统一异常处理、基础操作日志。

**交付物**：

1. Spring Boot 后端项目和 Vue 3 前端项目。
2. 登录页、主布局、项目列表、项目详情、成员管理页面。
3. 用户、角色、项目、项目成员相关表结构。
4. 项目管理接口和权限校验。
5. 基础开发、测试、生产配置文件。

**验收重点**：

1. 管理员可以登录并创建项目。
2. 项目负责人可以添加项目成员。
3. 非项目成员无法访问项目资产。
4. 项目归档后不出现在新建任务的项目选择项中。
5. 所有写操作能记录创建人或更新人。

**进入下一阶段条件**：

1. 权限模型稳定，不阻塞脚本和任务数据挂靠。
2. 项目资产可以作为后续所有业务数据的归属入口。
3. 前后端工程可以稳定启动、构建和部署。

### Phase 2：脚本管理与单机执行闭环

**目标**：实现从上传 JMX 到执行完成的核心压测流程。

**功能范围**：

1. JMX 上传、文件校验、脚本版本管理。
2. 脚本默认参数维护。
3. 测试任务创建、执行配置快照、单机队列。
4. JMeter 启动、执行、停止、结果文件归档。
5. 任务状态查询、执行日志查看、任务取消。
6. 平台重启后的运行中任务状态校准。

**技术重点**：

1. `FileStorage` 本地文件系统实现。
2. `TestExecutor` JMeter API 或 CLI 适配器实现。
3. `TaskScheduler` 基于数据库状态和 Spring Async 的单机实现。
4. 执行目录隔离：脚本、日志、结果、报告按项目和任务分目录。
5. 执行配置以 JSON 快照固化，避免脚本参数变更影响历史任务。

**交付物**：

1. JMX 上传、版本管理、参数配置。
2. 测试任务提交和单机队列。
3. JMeter API 集成。
4. 任务状态查询、执行日志查看、任务取消。
5. 平台重启后的任务状态校准。

**验收重点**：

1. 用户可以上传 JMX 并生成脚本版本。
2. 用户可以提交任务并看到状态变化。
3. 任务结束后保留日志和结果文件。
4. 用户可以取消待执行或运行中的任务。
5. 平台重启后，原运行中任务不会永久停留在运行中状态。

**进入下一阶段条件**：

1. 至少能稳定执行 3 个不同 JMX 示例脚本。
2. 成功、失败、取消、异常中断四类结果都有可追溯记录。
3. 执行结果文件结构满足报告生成需要。

### Phase 3：报告与结果可视化

**目标**：形成可交付的基础测试报告。

**功能范围**：

1. JMeter 结果文件解析和指标聚合。
2. 任务级、接口级统计指标展示。
3. TPS、响应时间、错误率、采样数趋势图。
4. Markdown 和 HTML 报告生成、查看、下载。
5. 报告与项目、脚本版本、执行配置、执行结果关联。

**技术重点**：

1. `ReportGenerator` Markdown/HTML 实现。
2. ECharts 图表组件封装。
3. 大结果文件异步报告生成，避免请求超时。
4. 报告状态管理：生成中、成功、失败。
5. 报告文件和原始结果文件分开存储。

**交付物**：

1. JMeter 结果解析。
2. TPS、响应时间、错误率、采样统计。
3. Markdown 和 HTML 报告生成。
4. 报告详情页、下载、项目内分享。
5. 基础图表展示。

**验收重点**：

1. 已结束任务可以生成报告。
2. 报告可追溯到项目、脚本版本和执行配置。
3. 项目成员可以查看报告，非成员不可查看。
4. 报告至少包含 TPS、平均响应时间、P90、P95、P99、错误率和采样总数。
5. 报告生成失败时可以查看失败原因并重新生成。

**进入下一阶段条件**：

1. 核心闭环已可用于真实项目试运行。
2. 报告指标得到测试负责人认可。
3. 用户开始提出监控、对比、导出等增强需求。

### Phase 4：监控采集与报告增强

**目标**：把压测结果和被测服务器资源指标关联起来，支持更完整的性能分析。

**功能范围**：

1. 监控目标管理，包括服务器名称、IP、标签、认证方式和项目归属。
2. CPU、内存、磁盘、网络等基础指标采集。
3. 任务执行期间自动关联监控目标和指标时间范围。
4. 报告中展示资源趋势图和关键时间点。
5. 支持 Word/PDF 导出和多次报告对比。

**技术重点**：

1. 先实现自研轻量 Agent 或 SSH 采集，避免一开始强依赖 Prometheus。
2. 指标数据按时间范围查询，保留后续接入 Prometheus 的适配层。
3. 报告导出通过模板化能力实现，避免和基础报告逻辑耦合。
4. 指标采样频率、保留周期和存储量需要可配置。

**交付物**：

1. 监控目标管理页面和接口。
2. 监控指标采集、存储和查询能力。
3. 任务报告中的资源趋势图。
4. Word/PDF 导出。
5. 两次测试报告对比页面。

**验收重点**：

1. 用户可以为项目配置监控目标。
2. 压测任务报告能展示执行期间的 CPU、内存、网络趋势。
3. 同一项目下两次报告可以对比 TPS、响应时间和错误率。
4. Word/PDF 文件可以下载并保留核心指标和图表。

**进入下一阶段条件**：

1. 监控数据采集不会明显影响被测服务器。
2. 报告增强能力已在真实压测复盘中使用。
3. 用户开始需要更高效的数据准备和脚本复用能力。

### Phase 5：造数工厂、函数库与辅助脚本

**目标**：提升测试准备效率，把重复的数据准备、函数封装和环境处理沉淀为平台资产。

**功能范围**：

1. 数据模板、字段定义、生成规则管理。
2. CSV、JSON、SQL 格式数据导出。
3. Groovy 自定义函数管理、分类、测试和调试。
4. 前置/后置辅助脚本管理。
5. 辅助脚本与测试任务关联。

**技术重点**：

1. 数据生成器接口支持内置规则和自定义规则。
2. Groovy 脚本执行需要沙箱、超时和权限限制。
3. Monaco Editor 用于函数和辅助脚本编辑。
4. SSH 远程执行能力必须记录执行日志和退出码。
5. 数据模板、函数、辅助脚本均归属项目或系统公共库。

**交付物**：

1. 造数模板管理和数据生成页面。
2. 函数库管理、在线调试和版本记录。
3. 辅助脚本管理和执行日志。
4. 测试任务前置/后置脚本配置。

**验收重点**：

1. 用户可以按模板生成测试数据并导出。
2. 用户可以编写并测试 Groovy 函数。
3. 测试任务可以关联前置或后置脚本，并记录执行结果。
4. 脚本执行失败时，测试任务能按配置停止或继续。

**进入下一阶段条件**：

1. 测试准备资产已有复用价值。
2. 用户需要把压测结果和代码、日志、变更关联分析。
3. 平台已有足够历史执行和报告数据供分析使用。

### Phase 6：Git、日志与 AI 分析

**目标**：从“执行和报告平台”升级为“性能问题辅助定位平台”。

**功能范围**：

1. Git 仓库配置、代码拉取、提交记录查看。
2. 测试任务关联代码分支、提交号、版本备注。
3. 日志上传、索引、检索和错误摘要。
4. AI 模型配置、Prompt 模板和分析任务。
5. 基于报告、监控、日志和代码变更生成优化建议。

**技术重点**：

1. JGit 作为默认 Git 客户端实现，保留 GitLab/GitHub API 扩展点。
2. 日志分析先采用 Lucene 或本地索引，数据量增长后再考虑 Elasticsearch。
3. AI 通过 `AIAnalyzer` 多模型适配器隔离供应商差异。
4. AI 分析必须保留输入数据、模型、Prompt 版本和输出结果，便于追溯。
5. 敏感信息进入 AI 前需要脱敏或显式确认。

**交付物**：

1. Git 仓库管理和提交记录页面。
2. 任务与代码版本关联。
3. 日志上传、检索和错误摘要。
4. AI 模型配置和分析记录。
5. 性能瓶颈分析和优化建议报告。

**验收重点**：

1. 用户可以将一次压测关联到代码提交。
2. 用户可以检索任务相关日志并定位错误片段。
3. AI 分析结果能引用具体指标、日志片段或代码变更依据。
4. AI 输出可追溯，不覆盖人工结论。

**进入下一阶段条件**：

1. 单机执行能力成为瓶颈，或存在多任务并发需求。
2. 多项目并发使用导致调度、资源隔离和扩展性问题。
3. 团队已准备好维护额外执行节点和中间件。

### Phase 7：执行规模扩展与平台化治理

**目标**：支持更高并发、更大规模的压测执行，并补齐平台运维治理能力。

**功能范围**：

1. 执行节点注册、心跳、标签和容量管理。
2. 分布式任务调度、任务分片、结果聚合。
3. Redis、RabbitMQ 或 Kafka 等中间件接入。
4. 多节点 JMeter 执行和结果回收。
5. 平台运行监控、告警、限流、配额和清理策略。

**技术重点**：

1. `TaskScheduler` 从单机实现切换为队列实现。
2. `TestExecutor` 支持远程执行节点。
3. 结果聚合需要处理节点失败、部分成功和重复上报。
4. 文件存储从本地文件系统升级为 MinIO 或 OSS。
5. 平台自身指标接入 Prometheus 和 Grafana。

**交付物**：

1. 执行节点管理页面。
2. 分布式调度和节点执行 Agent。
3. 分布式结果聚合和报告生成。
4. 平台资源配额、任务限流和数据清理策略。
5. 平台自身监控和告警。

**验收重点**：

1. 多个执行节点可以注册并接收任务。
2. 单个节点故障不会导致整个平台不可用。
3. 分布式任务结果可以聚合成一份报告。
4. 管理员可以限制项目并发数和单任务资源占用。

### 7.2 阶段推进建议

1. Phase 0 到 Phase 3 必须连续推进，这是平台能否使用的最小路径。
2. Phase 4 到 Phase 6 可以根据用户反馈调整顺序；如果用户更关注数据准备，可以先做 Phase 5。
3. Phase 7 不应提前启动，除非单机执行已经明确成为瓶颈。
4. 每个阶段结束后都要沉淀一次真实项目试运行报告，用它决定下一阶段优先级。
5. 技术上始终保持接口抽象，但实现上只引入当前阶段真正需要的组件。

## 八、风险与约束

| 风险 | 影响 | MVP 处理策略 |
|------|------|--------------|
| JMeter 执行占用服务器资源过高 | 平台不稳定，影响其他任务 | 限制并发任务数，记录进程和资源配置 |
| 用户上传恶意或异常 JMX | 文件安全和执行风险 | 限制文件类型、大小、存储目录，执行环境隔离 |
| 长任务状态不一致 | 页面显示和实际进程不一致 | 状态落库，启动时校准运行中任务 |
| 结果文件过大 | 报告生成慢或内存溢出 | 异步生成报告，限制单次解析内存使用 |
| 取消任务失败 | 资源持续占用 | 记录失败原因，提供异常中断状态 |
| 权限边界不清晰 | 测试资产误访问 | MVP 采用系统角色 + 项目成员双重校验 |
| 首版范围膨胀 | 延期且核心流程不稳定 | 监控、AI、造数、Git、分布式均放入后续阶段 |

## 九、关键技术决策

| 决策点 | 方案 | 理由 |
|--------|------|------|
| 架构风格 | 单体分层 + 模块化 | 小团队内部平台优先降低部署和协作成本 |
| 后端语言 | Java | 与 JMeter 同生态，便于直接集成 JMeter API |
| 前端框架 | Vue 3 + TypeScript | 适合企业后台和复杂表单页面 |
| 数据库 | H2 开发 / MySQL 生产 | 开发轻量，生产稳定 |
| 任务调度 | Spring Async + 数据库任务表 | 满足 MVP 单机队列，不引入 MQ |
| 文件存储 | 本地文件系统 | 支撑 JMX、日志、结果和报告的低成本存储 |
| 报告格式 | Markdown + HTML | 首版易生成、易查看、易下载 |
| 监控方案 | 后续自研 Agent 或 Prometheus 生态 | 不进入 MVP，避免执行闭环被监控复杂度阻塞 |
| AI 集成 | 后续多模型适配器 | 基于稳定报告和日志数据再建设 |

## 十、验收与评审清单

### 10.1 文档结构检查

1. 章节编号从一到十连续。
2. 标题 Markdown 语法闭合正确。
3. 不再出现“8 个核心领域但列出 10 个领域”的矛盾描述。
4. 所有 Phase 标题格式统一。

### 10.2 范围检查

1. MVP 只包含项目管理、脚本管理、测试执行、报告管理、系统管理。
2. 监控、AI、造数、函数库、Git、辅助脚本、分布式执行均不作为 MVP 必做。
3. 技术选型能对应到具体需求，不单独堆砌扩展能力。

### 10.3 需求可验收检查

1. 每个 MVP 模块至少包含一个 `REQ-模块-序号` 需求。
2. 每条需求包含用户故事、功能说明、输入/输出、异常场景和验收标准。
3. 验收标准可以通过页面操作、接口返回或数据记录验证。

### 10.4 场景走查

按以下路径评审文档是否闭环：

```text
管理员创建用户和项目
-> 项目负责人添加成员
-> 性能测试工程师上传 JMX
-> 保存脚本参数
-> 提交测试任务
-> 查看执行状态和日志
-> 任务结束后生成报告
-> 项目成员查看或下载报告
```

如果任一步无法从需求、数据模型或架构说明中找到支撑内容，则该文档未通过评审。
