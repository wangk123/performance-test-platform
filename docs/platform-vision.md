## 二、脚本创建 — 三种入口

### 2.1 JMX 导入 + 可视化编辑（无损）

**现状问题**：当前 Parser 只识别 6 种 JMX 元素（HTTPSamplerProxy / CSVDataSet / Arguments / HeaderManager / ResponseAssertion / ThreadGroup），其余元素在 `default → {}` 分支静默丢弃。保存时 Renderer 全量重新生成 XML，导致不支持的元件永久丢失。

**目标**：导入任意 .jmx，可视化编辑器只渲染可识别部分，不可识别部分在 DOM 中原样保留。保存时局部 patch 变更，不动未识别节点。

```
核心变化:

  当前（有损）:
    原始 XML → Parser 全量解析 → ScriptStepDefinition[] → Renderer 全量生成 → 新 XML
    Parser 不认识的部分 → 永久丢失

  目标（无损）:
    原始 XML → w3c DOM 解析 → 提取可识别节点(保留 DOM 引用/路径)
           → ScriptStepDefinition[] 给前端渲染
           → 前端编辑后返回变更列表
           → 定位原 DOM 节点 → 局部 patch → 写回
    不可识别节点 → 完全不动
```

**实现要点**：
- Parser 为每个识别出的步骤记录在原始 DOM 中的定位信息（XPath 或节点引用）
- 渲染回写时，对已识别步骤执行局部替换或新增；未识别节点完全透传
- 新增步骤（手动创建/AI 生成）插入到对应 ThreadGroup 的 hashTree 末尾

**可识别的元素范围**：不追求覆盖 JMeter 全集。只覆盖 HTTP API 场景的常用元件，其余均保留不处理。后续按需求逐步扩展识别范围。

### 2.2 AI 生成脚本

**输入一：接口文档（OpenAPI / Swagger JSON）**
```
OpenAPI JSON → 遍历 paths → 提取 method/url/params/body schema
           → 组装 ScriptStepDefinition[] → 前端渲染
```

**输入二：自然语言描述**
```
用户描述 → LLM 调用 → 结构化 ScriptStepDefinition JSON → 前端渲染
```

**输出**：生成的脚本进入可视化编辑器，用户可二次调整。

**参考**：PerfAI（OpenAPI → JMX 生成）、k6 社区的 AI 脚本生成实践。

### 2.3 手动创建脚本

从空步骤树开始，逐条添加步骤：
- 添加 HTTP 请求（配置 URL/Method/Body/Headers）
- 添加响应断言（状态码/响应体/JSON Path）
- 添加 JSON 提取器（提取 token 等动态值）
- 添加 CSV 数据文件（参数化）
- 添加 Header 头配置
- 配置线程组参数

当前 `ScriptStepDefinition` 模型已支持，前端组件部分就位。

## 三、XML 双模切换

可视化编辑器和原始 XML 编辑器并存于同一视图，通过 Tab 或按钮切换。

```
┌─────────────────────────────────────────────┐
│  [可视化模式]  [XML 模式]                    │
├─────────────────────────────────────────────┤
│                                             │
│   （步骤树 / 表单 / 拖拽编辑）               │
│                                             │
│   或                                         │
│                                             │
│   （Monaco Editor / 语法高亮 / 代码补全）     │
│                                             │
└─────────────────────────────────────────────┘
```

**目标**：
- 可视化模式：覆盖常用场景的快速编辑，降低使用门槛
- XML 模式：高级用户直接编辑源码，支持跨脚本复制粘贴
- 切换时保持双向同步（切到 XML 时从 DOM 渲染一次，切回可视化时重新解析）
- 支持下载导出 .jmx，在 JMeter GUI 中编辑后重新上传覆盖

**实现**：前端集成 Monaco Editor，后端提供 XML Content API（已有 `ScriptContent` 接口）。

## 四、多引擎导出

不改变主存储格式（仍为 .jmx），引擎导出为"生成"操作：

```
.jmx 脚本（主存储格式）
      │
      ├── JMeter CLI 直接执行（原生支持，无转换）
      │
      ├── 导出为 k6 .js 脚本
      │    └── 映射：HTTPSampler→http.get/post()、ThreadGroup→options.stages
      │               Assertion→check()、CSVDataSet→SharedArray
      │
      └── 导出为 Gatling .scala 脚本
           └── 映射：HTTPSampler→http("name").get/post()
                     ThreadGroup→setUp().inject()
                     Assertion→.check(status.is(200))
```

**定位**：导出是有损的——JMeter 特有协议（JDBC/JMS/FTP 等）无法转换。导出前提示用户哪些步骤无法映射，由用户确认。

**导出不是主存储格式的迁移，而是"当前脚本多引擎执行"的便捷通道。**

## 五、远程 Agent 执行

> 核心需求：脚本执行不落在平台服务器上，下发到与被测系统同网段的 Agent 节点执行。

```
  管理平台 (Spring Boot)           Agent 节点 A (同网段)       Agent 节点 B (同网段)
  ┌──────────────────┐          ┌──────────────────┐        ┌──────────────────┐
  │  脚本管理          │          │  任务接收          │        │  任务接收          │
  │  任务编排          │  HTTP/gRPC │  脚本拉取          │        │  脚本拉取          │
  │  结果展示          │─────────▶│  JMeter CLI 执行   │        │  k6 CLI 执行      │
  │  Agent 注册/心跳    │◀─────────│  结果上报          │        │  结果上报          │
  └──────────────────┘          └────────┬─────────┘        └────────┬─────────┘
                                         │                           │
                                         ▼                           ▼
                                   ┌──────────┐              ┌──────────┐
                                   │ 被测系统A  │              │ 被测系统B  │
                                   └──────────┘              └──────────┘
```

**关键点**：
- Agent 注册时绑定环境标签（SIT/UAT/PRD），平台按标签自动匹配
- 脚本由平台统一管理，Agent 按需下载到临时目录
- 执行完成后结果回传到平台存储
- Agent 心跳超时自动标记离线，已下发任务在 Agent 本地继续执行
- 同一 Agent 并发执行数可配置上限

详见 `docs/performance-platform-comparison.md` 第 8.5 节。
