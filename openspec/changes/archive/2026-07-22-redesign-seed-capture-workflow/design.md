## Context

现有录制以 `seed_capture_session` 表示一次长生命周期会话。创建会话时同步读取所有目标表作为基线，用户结束样本时再次同步读取并立即计算 Diff，结束会话时自动推断模板。前端只保留内存中的会话 ID，没有历史查询、进度、详情和分页能力；完整快照与样本 Diff 以整块 LONGTEXT JSON 保存，无法支撑半小时级、大体量、多次录制。

本变更面向单实例后端和持久化本地磁盘。样本和分析结果默认永久保留，仅允许用户手动清理；不引入对象存储或新的文件格式依赖。

## Goals / Non-Goals

**Goals:**

- 将录制配置、样本执行和多样本分析拆成独立、可查询的领域对象。
- 支持同一策略无限次手动录制，并以不可变采集时间建立稳定顺序。
- 以有界内存和可配置并发采集大表，持续展示真实进度并允许取消。
- 以数据库元数据和本地压缩分片组合存储大样本，支持服务端分页详情。
- 对至少三份快照执行相邻 Diff，并通过粗筛减少无变化表和分片的读取。
- 永久保留样本、分析记录和模板，同时保证清理、宕机和磁盘不足时状态可解释。

**Non-Goals:**

- BINLOG/CDC 采集、生产库写入、DELETE 克隆、业务 API 回放。
- 多实例任务调度、共享对象存储、跨节点文件访问。
- 运行中断后的自动续跑或自动从头重跑。
- 自动清理、保留周期或存储配额分摊。
- 无主键表的可靠 UPDATE 身份识别和自动可执行克隆。

## Decisions

### D1: 使用“策略 → 样本 → 分析 → 模板”的领域模型

- `seed_capture_strategy` 保存数据源、include/exclude、线程数、每批行数及版本化配置。
- `seed_capture_sample` 表示一次手动执行，创建时复制策略配置并记录 `capture_started_at` 与策略内递增 `sample_seq`。
- `seed_capture_table` 和 `seed_capture_chunk` 保存样本的库表、分片、进度、指纹和文件清单。
- `seed_capture_analysis` 保存一次手动多样本分析的状态、输入快照、摘要和结果清单；运行期间用输入锁阻止样本删除。
- 模板草稿复制分析产物所需数据，不依赖原样本继续存在。

选择该模型是为了让配置可复用、执行可审计、结果可独立清理。继续扩展单个 session JSON 会保留同步耦合和整块读取问题，因此不采用。

### D2: 平台数据库保存控制面，本地分片文件保存数据面

数据库保存 ID、状态、时间、进度、表结构、行数、相对路径、主键范围、逻辑指纹、文件 checksum 和分析摘要。快照行与详细 Diff 写入：

`storage/seed-captures/project-{projectId}/strategy-{strategyId}/sample-{sampleId}/{table}/chunk-{seq}.jsonl.gz`

分析详情使用同类的 analysis 目录。路径以 `sample_id` / `analysis_id` 为稳定锚点，数据库只保存相对路径。JSONL.GZIP 可复用现有 JSON 能力与 JDK GZIP，支持流式读写且不新增依赖；相比数据库逐行写入，顺序文件写入减少事务日志、索引和行存储开销。

### D3: 分片以临时写入、校验和原子改名完成提交

Worker 将一批行写入 `.tmp`，完成 flush 后计算文件 checksum，再原子改名为 `.jsonl.gz`，最后将数据库 manifest 标记为 `READY`。文件 checksum 只校验物理文件完整性，不参与业务数据是否变化的判断。失败或取消时删除未提交临时文件，保留已 READY 分片。

### D4: 同一数据源串行录制，单次录制内部并行

创建样本前获取 datasource 级持久化租约；已有活跃录制时拒绝新执行并返回活跃样本。每次录制使用策略配置的固定 Worker Pool：

- 默认按表创建工作单元。
- 单列数值主键大表可按稳定 PK 范围拆分。
- 复合或字符串主键使用 keyset 分批。
- 无主键表单流读取并标记风险。

`batchRows` 控制单次查询和文件分片的最大行数，线程数控制同时运行的 Worker。录制不额外执行精确 `COUNT(*)`；进度以阶段、已完成表数/总表数为主，同时展示当前表、已采集行数、已写字节和活动 Worker。

### D5: 采集开始时间定义样本顺序

样本创建时由服务端写入不可变 UTC 高精度 `capture_started_at`；`capture_finished_at` 仅表示完成时间。列表默认按采集开始时间倒序，分析按 `capture_started_at ASC, sample_seq ASC` 排序，`sample_seq` 解决时间相同和时钟精度不足时的稳定排序。

### D6: 录制时生成逻辑指纹，Diff 时三级筛选

采集行在序列化前先按字段类型、NULL、长度和值做规范化编码，并在写分片时计算逻辑 SHA-256 指纹。manifest 保存分片级 `rowCount + contentHash` 和表级 `schemaHash + rowCount + contentHash`。

相邻样本分析按以下顺序执行：

1. 表级 manifest 相同则标记 `UNCHANGED`，不打开快照文件。
2. 表发生变化时，比较兼容且稳定的逻辑分片指纹，跳过相同主键范围。
3. 只对变化分片执行主键有序归并和字段级比较。

策略版本、表结构、指纹缺失或数据不完整时禁止用“相同”结论静默跳过，必须进入细筛或标记 `UNKNOWN`。无主键表使用顺序无关的全行 hash 多重集指纹；表变化后只能识别集合增减，不能可靠识别 UPDATE 身份。

### D7: 至少三份快照按相邻区间生成 Diff

分析必须选择同一策略的至少三份终态样本。排序后的 `S1...Sn` 生成 `n-1` 份 Diff：`Δi = S(i+1) - Si`。S1 是第一段基线，之后每份快照自动成为下一段基线。选择的 `sample_seq` 不连续时展示累计变化风险，但允许继续。

有主键表识别 INSERT、UPDATE；DELETE 仅保留诊断信息，不进入模板。失败、取消或中断样本允许在显式确认后参与分析；缺表、缺分片和校验失败只能产生 `UNKNOWN`，不得解释为空表、无变化或 DELETE。

### D8: 录制和分析均为可取消的异步状态机

录制状态：`QUEUED → PREPARING → CAPTURING → SUCCEEDED`，取消路径为 `CANCEL_REQUESTED → CANCELED`，执行态可进入 `FAILED` 或 `INTERRUPTED`。

分析状态：`QUEUED → VALIDATING → DIFFING → INFERRING → PERSISTING → SUCCEEDED`，并具有同类取消、失败和中断终态。取消采用协作式检查，Worker 完成当前批后停止。

运行任务定期刷新心跳和租约。服务启动时将心跳过期的运行态记录改为 `INTERRUPTED`、释放租约与样本锁并保留已提交分片；用户自行删除记录并重新执行，不自动续跑。

### D9: 永久保留与手动删除相结合

样本、分析记录和模板默认不自动清理。分析运行期间，所选样本禁止删除；分析进入终态后解除输入锁，分析输入 manifest 和输出自包含，样本、分析与模板可分别删除且不级联。

删除采用 `DELETING` 中间态：先阻止新引用，再删除文件与分片清单，最后删除元数据；清理失败保留状态并允许重试。旧 `seed_capture_session` 数据不迁移。

### D10: 增加磁盘低水位保护

录制和分析开始前检查持久化目录可用空间；低于可配置阈值时拒绝执行。运行中定期检查，触发低水位时停止任务并标记失败，保留已提交分片，绝不自动删除历史样本。

### D11: 所有大数据详情均服务端分页

策略、样本和分析记录使用数据库分页。样本表行与分析行级 Diff 使用 `cursor + limit` 定位分片，默认小页读取；接口不得反序列化整表或整份样本。详情先返回库表摘要，用户展开表后才读取目标分片。

## Risks / Trade-offs

- [本地磁盘成为单点且容量有限] → 明确单实例约束、增加低水位保护、校验与可重试清理；未来可在不改变领域模型的前提下替换文件后端。
- [源库扫描影响测试环境] → include 必填、同一数据源串行、线程数和每批行数可控，不额外执行精确 COUNT。
- [并行扫描不是全库同一时刻快照] → 将 `capture_started_at` 定义为样本时刻并接受时间窗口语义，不锁表。
- [逻辑指纹实现错误导致漏 Diff] → 使用稳定规范化编码、SHA-256、行数与 schemaHash 组合，并用全量 Diff 对照测试验证；未知或不兼容情况不得跳过。
- [相邻但不连续样本包含累计业务变化] → 在分析前展示序号缺口和时间范围风险，由用户显式确认。
- [异常样本影响推断] → 保留覆盖率和 UNKNOWN，模板候选展示依据与风险，禁止把缺失数据当作无变化。
- [大量小文件增加管理成本] → 每批行数控制分片数量，manifest 提供定向访问，删除按样本目录聚合执行。
- [旧 API 破坏兼容] → 作为明确 breaking change 同步切换前后端，不提供双轨写入。

## Migration Plan

1. 新增策略、样本、表/分片、分析和锁/租约数据结构，以及本地存储目录配置。
2. 停止旧录制入口并清理旧 `seed_capture_session` 数据；不做格式转换。
3. 上线新异步录制、查询、分页、取消、清理和分析 API。
4. 切换造数工厂录制页到策略与样本工作流，再启用手动分析和模板草稿入口。
5. 启动时清理孤立 `.tmp` 文件并执行过期心跳协调。

回滚时停用新 UI/API 并恢复旧版本代码；新增表和已生成文件暂时保留，避免回滚过程再次破坏数据。旧会话数据已清理，无法通过回滚恢复。

## Verification Strategy

- 单元测试覆盖状态机、时间排序、规范化编码、指纹稳定性、表/分片粗筛、相邻 Diff、无主键多重集和 UNKNOWN 传播。
- 集成测试覆盖 datasource 租约、异步进度、取消、宕机协调、临时文件原子提交、checksum、磁盘低水位、分页和删除锁。
- 使用相同数据分别运行“三级筛选 Diff”和“直接全量 Diff”，断言结果一致。
- 使用大表与大量未变化表验证内存有界、未变化表不打开数据文件、分页仅读取目标分片。
- 前端验证策略保存不执行、长任务状态恢复、异常样本风险确认、至少三份选择门禁和详情懒加载。
